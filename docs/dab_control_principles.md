# DAB 控制原理说明文档

## 0. 前提和诚实声明

本文默认 **DAB = Dual Active Bridge，双有源桥隔离型双向 DC/DC**。

这是一个**明确假设**，不是从当前仓库源码里直接读到的既有定义。当前工程里没有 `DAB` 命名的模块、变量或注释，但存在以下和“双母线/第二桥级”强相关的证据：

1. 原始整机工程的 `src/gridvoltage.h` 中定义了 `vdc1`、`vdc2`。
2. 原始整机工程的 `src/gcc.h` 中存在 `Vdc1`、`Vdc2`、`Vdc2RefInit` 以及模块 2 相关的预留字段。
3. 原始整机工程的 `src/gcc_dspa.c` / `src/gcc_dspb.c` 共享内存读写里，`0x24 ~ 0x28` 一段对“第二路 Vdc/电流校准”的代码是预留或被注释掉的。
4. [../src/current_loop.c](../src/current_loop.c) 和 [../src/Vbus_volt_ctrl.c](../src/Vbus_volt_ctrl.c) 只真正闭环了**单个并网侧直流母线**，没有第二功率级控制抽象。

因此，本文把“增加 DAB 控制”解释为：

**在现有 SVG 并网变流控制之外，新增一个独立的、模块化的、面向双母线能量转移的 DAB 控制链路。**

如果你这里的 DAB 不是双有源桥，而是别的缩写，这份文档里的“模块拆分原则、边界划分原则、状态机/保护/调度设计”仍然能复用，但功率方程和调制部分需要重写。

## 1. 先看清当前工程到底已经有什么

### 1.1 原始工程的控制事实

当前代码里已经闭环运行的是一条很清晰的主链路：

1. 采样三相电压、负载电流、SVG 输出电流、`Vdc1`
2. PLL 锁相，得到基波角度
3. 从负载电流里分离无功、谐波、负序分量
4. 用 `Vdc1` 电压环生成有功分量
5. 合成三相补偿电流参考
6. 电流 PI 输出三相调制量
7. 送给 PWM

这套链路的本质是：

**并网侧三相变流器负责把交流侧能量和 `Vdc1` 母线能量维持住。**

### 1.2 当前仓库的第二母线状态

仓库里出现了 `Vdc2`，但没有形成完整控制链：

1. 有变量定义。
2. 有部分共享内存地址预留。
3. 没看到 `Vdc2` 被真正纳入闭环控制。
4. 没看到“第二桥级 PWM 输出”“移相调制结果”“桥臂状态机”这类完整 DAB 行为。

这说明当前工程更像是：

**为第二母线/第二功率级留了接口，但没把控制闭环搭完整。**

### 1.3 现有 `control_algo_lib` 的边界

[../README.md](../README.md) 已经把边界定义得很清楚：

1. 纯算法可以进库。
2. 芯片寄存器、PWM 外设、ADC 通道、共享内存地址、Modbus 映射不能直接进库。
3. 一个功能一个 `.h/.c` 模块。

所以 DAB 也必须遵守同样边界：

**DAB 的控制规律、状态机、限幅、功率解算、相移求解可以进算法库；具体桥臂 PWM 映射、硬件死区、互锁、采样通道号、驱动故障引脚必须留在应用层/适配层。**

## 2. 为什么 DAB 不能直接塞进现有 `current_loop`

直觉上最容易犯的错，是把 DAB 当成“再多一个控制分量”，直接和基波/谐波电流控制混在一起。

这在架构上是错的，原因有三层。

### 2.1 DAB 控的不是三相补偿电流

现有 `current_loop` 的输出是：

1. 三相电压指令 `abc`
2. 三相占空比

而 DAB 真正的控制输出通常不是三相 `abc` 电压，而是：

1. 主副桥之间的相移角
2. 或扩展调制参数（SPS / DPS / TPS）
3. 或桥臂时序

也就是说，**输出空间都不一样**。

### 2.2 DAB 面向的是双母线能量转移，不是并网补偿

并网 SVG 控制关注：

1. 无功补偿
2. 谐波补偿
3. 并网电流成形
4. `Vdc1` 稳压

DAB 控制关注：

1. `Vdc1 -> Vdc2` 或 `Vdc2 -> Vdc1` 的双向功率流
2. 隔离变压器漏感电流约束
3. 主副桥电压比变化下的功率可达性
4. 软启动、预充、相移限幅、零功率区切换

它们耦合，但不是同一个控制对象。

### 2.3 如果不分层，会让两个母线环互相打架

假设 DAB 直接根据 `Vdc2` 误差猛拉功率，而并网侧 `Vdc1` 电压环也在独立积分：

1. `Vdc2` 低，DAB 增大相移，开始从 `Vdc1` 抽能。
2. `Vdc1` 被抽低，并网侧 `Vdc1` 电压环再提高有功电流参考。
3. 如果两边带宽、限幅、启动逻辑没分清，就会出现：
   1. `Vdc1` 振荡
   2. `Vdc2` 振荡
   3. DAB 方向来回抖
   4. 软启动时直流侧冲击

所以必须把 DAB 定义为**独立功率级控制模块**，并通过清晰的功率边界与并网侧协同。

## 3. DAB 的物理原理

## 3.1 拓扑本质

双有源桥的基本拓扑是：

1. 一次侧一个全桥
2. 二次侧一个全桥
3. 中间一个高频变压器
4. 变压器漏感或外串联电感作为能量传递电感

两个桥都把各自直流母线变成高频方波。

当两边方波存在相位差时，漏感上产生电压差，进而形成电流，完成能量从一侧母线到另一侧母线的传递。

### 3.2 为什么“相移”能决定功率

先忽略器件损耗、磁化支路和死区，只看最常见的单移相 SPS：

1. 一次侧桥输出幅值约等于 `±V1`
2. 二次侧折算到一次侧后幅值约等于 `±n * V2`
3. 两个方波如果同相，漏感两端平均做功接近 0
4. 两个方波如果有相移 `phi`，漏感上会出现分段电压差
5. 漏感电流按分段线性规律上升/下降
6. 一个周期平均功率由 `V1`、`V2`、`Lk`、开关角频率、相移角共同决定

直觉上可以理解成：

**相移越大，两边桥在“推着电感送能量”的时间越长，平均传输功率越大。**

### 3.3 常用 SPS 功率近似式

在标准 SPS 近似下，平均功率常写成：

```text
P ≈ (n * V1 * V2 / (w * Lk)) * phi * (1 - |phi| / pi)
```

其中：

1. `V1`：一次侧母线电压
2. `V2`：二次侧母线电压
3. `n`：折算系数（按约定把二次侧折算到一次侧）
4. `w = 2 * pi * fs`
5. `Lk`：等效漏感
6. `phi`：相移角，通常范围限制在 `[-pi/2, +pi/2]` 或更小

这个公式告诉我们几件事：

1. 当 `phi = 0`，平均功率接近 0。
2. `phi` 正负决定功率方向。
3. 对同样的相移，`V1`、`V2` 越高，可传功率越大。
4. `Lk` 越大，功率变化越“钝”；`Lk` 越小，功率变化越“猛”。

### 3.4 为什么不能只靠这个公式直接“裸奔”

因为真实工程里还有这些影响：

1. 死区
2. 器件压降
3. 采样延迟
4. 电压比变化
5. 磁化电流
6. 漏感建模偏差
7. 轻载区零电流附近的方向抖动
8. 大相移区功率非线性更强

所以工程上不能只有“功率公式 -> 相移”这一层，还必须叠加：

1. 外环目标生成
2. 限幅
3. 斜率限制
4. 软启动
5. 故障保护
6. 状态机

## 4. 本项目里 DAB 的推荐职责

## 4.1 推荐控制目标

在这个 SVG 工程上下文里，最合理的第一版 DAB 目标不是“什么都做”，而是先做成：

**维持 `Vdc2`，并在 `Vdc1` 与 `Vdc2` 之间安全、可限幅、可方向切换地传递功率。**

也就是：

1. 并网三相变流器继续负责 `Vdc1`
2. DAB 负责 `Vdc2`
3. `Vdc1` 是 DAB 的能量来源或回灌去向

### 4.2 DAB 不该负责什么

第一版 DAB 不建议直接负责：

1. 三相无功补偿
2. 谐波补偿
3. 三相 PWM 占空比生成
4. Modbus 地址表编排
5. GPIO/继电器动作
6. DSPA/DSPB 共享内存搬运

这些都不是 DAB 纯算法内核本身。

### 4.3 DAB 和现有 `Vbus_volt_ctrl` 的关系

现有 `Vbus_volt_ctrl` 控的是并网桥直流母线。

新增 DAB 后，推荐关系是：

1. `Vbus_volt_ctrl`：维持 `Vdc1`
2. `dab_vdc_ctrl`：维持 `Vdc2`
3. DAB 把 `Vdc2` 误差变成功率需求
4. 这个功率需求最终通过相移体现为“从 `Vdc1` 取能”或“向 `Vdc1` 回能”

这里不是两个控制器直接互相调用，而是通过真实母线能量耦合。

## 5. 推荐的控制分层

为了把 DAB 拆清楚，推荐最少分 6 层。

## 5.1 层 1：输入归一化层

职责：

1. 接收 `Vdc1`
2. 接收 `Vdc2`
3. 接收 DAB 电流反馈（如果硬件有）
4. 接收使能、方向、参考值、限幅值
5. 把单位统一成算法库内部统一量纲

这一层不做控制计算，只负责“把外部世界喂给算法”。

## 5.2 层 2：模式/状态机层

职责：

1. 决定当前是 `STOP`、`READY`、`SOFTSTART`、`RUN` 还是 `FAULT`
2. 控制什么时候允许输出相移
3. 控制什么时候只能输出 0 相移
4. 处理方向切换前的归零
5. 处理故障后的闭锁

这是 DAB 最容易被忽略、但最不能省的一层。

### 为什么状态机必须单独成模块

因为 DAB 的危险，不在“公式算不出来”，而在：

1. 上电就给相移
2. 故障后还保留积分
3. 方向切换时跨零不平滑
4. 母线未就绪就进功率传输

这些都不是 PI 参数能解决的，是状态管理问题。

## 5.3 层 3：外环电压控制层

职责：

1. 计算 `Vdc2_ref - Vdc2_fdb`
2. 通过 PI 或 PID 生成 `P_ref`
3. 或生成等价的 `I_lk_ref`
4. 对参考做上限、下限、斜率限制

推荐第一版输出 `P_ref`，理由是：

1. 与 DAB 功率方程天然对应
2. 更容易和 `V1/V2` 前馈结合
3. 更适合作为系统级能量接口

### 为什么不建议第一版直接输出相移

如果电压环直接输出相移：

1. 不同 `V1/V2` 下同样相移代表的功率不同
2. 电压波动时环路等效增益漂移大
3. 参数整定会很难迁移

所以更推荐：

**电压环输出功率参考，功率参考再通过模型换成相移。**

## 5.4 层 4：功率前馈/归一化层

职责：

1. 使用 `V1`、`V2`、`Lk`、`fs` 估算当前功率增益
2. 把 `P_ref` 转成相移求解所需的归一化量
3. 做可达性裁剪

这是为了减轻外环的负担。

如果没有这一层，会出现：

1. `V1` 高时系统太敏感
2. `V1` 低时系统太迟钝
3. 同一套 PI 参数在不同母线条件下表现完全不一样

## 5.5 层 5：相移求解层

职责：

1. 根据 `P_ref` 和当前 `V1/V2` 求 `phi_ref`
2. 对 `phi_ref` 做限幅
3. 对 `phi_ref` 做斜率限制
4. 输出归一化相移 `phase_shift_pu`

### 第一版推荐的调制方式

第一版推荐只做 **SPS 单移相**，不建议一上来就上 DPS/TPS。

理由：

1. 现有仓库还在做算法模块化重构，先把边界走通更重要。
2. SPS 最容易形成稳定、可读、可调的第一版。
3. 只有当轻载效率、回流电流、ZVS 范围成为明确痛点时，再升级 DPS/TPS。

## 5.6 层 6：保护层

职责：

1. 母线过压/欠压
2. 相移饱和持续过久
3. 电流过限
4. 方向切换保护
5. 软启动超时
6. 模型参数非法

保护层的输出不是“继续努力控制”，而是：

**告诉状态机是否应该降额、限幅、归零或锁故障。**

## 6. 推荐的控制时序

## 6.1 快速环和慢速环必须分开

建议按两个时间尺度处理。

### 快速任务：每个 ISR 周期

做这些：

1. 采样值更新
2. 故障硬限判断
3. 相移输出斜率限制
4. 调制输出更新

### 慢速任务：每 `N` 个 ISR 周期

做这些：

1. `Vdc2` 电压环
2. 功率参考更新
3. 可达性裁剪
4. 软启动斜坡推进

原因是：

1. 电压环本来就不需要每个开关周期都积分
2. 相移求解可能涉及平方根或更复杂运算
3. 把慢环降采样后，ISR 时间更稳

## 6.2 建议带宽关系

推荐关系只给相对顺序，不强塞具体数值：

1. 保护判定最快
2. 相移输出更新次之
3. DAB 外环电压控制再慢一些
4. 系统级运行/模式切换最慢

如果未来要做 `Vdc1` 与 `Vdc2` 的双母线协同，建议：

**不要让 `Vdc1` 环和 `Vdc2` 环带宽相等。**

否则很容易互相追着打。

## 7. 推荐的模块拆分

下面是按“一个功能一个模块”的拆法。

## 7.1 纯算法层模块

### `dab_mode`

职责：

1. 维护 DAB 状态机
2. 处理使能、停机、故障、方向切换
3. 产生“当前是否允许输出”的判定

输入：

1. enable
2. ready 条件
3. 故障标志
4. 当前方向请求

输出：

1. state
2. output_enabled
3. softstart_active

### `dab_vdc_ctrl`

职责：

1. `Vdc2` 外环控制
2. 输出 `P_ref`
3. 做积分限幅

输入：

1. `Vdc2_ref`
2. `Vdc2_fdb`

输出：

1. `P_ref_raw`

### `dab_power_feedforward`

职责：

1. 根据 `V1/V2/fs/Lk/n` 计算当前功率增益
2. 把 `P_ref_raw` 变成可求解的归一化目标
3. 做功率可达性裁剪

输入：

1. `P_ref_raw`
2. `Vdc1`
3. `Vdc2`
4. 参数 `n/Lk/fs`

输出：

1. `P_ref_limited`
2. `power_gain`

### `dab_phase_shift`

职责：

1. `P_ref -> phi_ref`
2. `phi_ref` 限幅
3. `phi_ref` 斜率限制
4. 输出标准化相移

输入：

1. `P_ref_limited`
2. `power_gain`
3. 相移上下限

输出：

1. `phi_cmd`
2. `phase_shift_pu`

### `dab_softstart`

职责：

1. 启动时相移缓升
2. 方向切换前缓降到 0
3. 母线恢复后平滑重启

输入：

1. 目标相移
2. 当前状态

输出：

1. 实际允许输出的相移命令

### `dab_protection`

职责：

1. 过压/欠压/过流/参数错误检测
2. 输出降额、闭锁或故障字

输入：

1. `Vdc1`
2. `Vdc2`
3. DAB 电流反馈
4. 相移命令

输出：

1. warning
2. fault
3. derate_limit

### `dab_control`

职责：

1. 编排以上所有模块
2. 形成 DAB 总输出
3. 对外暴露统一 API

输入：

1. DAB 输入
2. DAB 配置
3. 各子模块状态

输出：

1. DAB 输出
2. DAB 状态/故障

## 7.2 必须留在应用层/适配层的模块

### `dab_pwm_adapter`

职责：

1. 把 `phase_shift_pu` 变成具体 ePWM 比较值
2. 处理桥臂互补、死区、同步触发
3. 处理芯片寄存器写入

不能进算法库的原因：

1. 强依赖 DSP281x/C2000 外设
2. 强依赖具体桥臂映射

### `dab_adc_adapter`

职责：

1. 读取 `Vdc1/Vdc2` 采样
2. 读取桥电流/漏感电流采样
3. 做硬件量到归一化量换算

### `dab_comm_adapter`

职责：

1. Modbus 寄存器映射
2. DSPA/DSPB 共享内存地址映射
3. 上位机调参/状态回读

## 9. 推荐状态机

## 9.1 状态定义

建议至少有这些状态：

1. `DAB_ST_STOP`
2. `DAB_ST_READY`
3. `DAB_ST_SOFTSTART`
4. `DAB_ST_RUN`
5. `DAB_ST_DERATING`
6. `DAB_ST_FAULT`

### 每个状态的含义

`DAB_ST_STOP`

1. 不允许输出相移
2. 积分器复位或冻结
3. 等待使能

`DAB_ST_READY`

1. 条件满足但还未真正送能
2. 可以做预充检查、母线有效性检查

`DAB_ST_SOFTSTART`

1. 从 0 相移平滑拉升
2. 不允许瞬间跳到目标相移

`DAB_ST_RUN`

1. 正常闭环运行
2. 外环、限幅、保护全部生效

`DAB_ST_DERATING`

1. 还没到故障，但已经需要降额
2. 例如电流逼近上限、母线余量不足

`DAB_ST_FAULT`

1. 输出强制归零
2. 等待人工复位或自动复位条件

## 9.2 关键状态转移规则

1. `STOP -> READY`
   条件：使能有效，参数合法，母线采样有效。

2. `READY -> SOFTSTART`
   条件：母线电压进入允许区间，方向请求稳定。

3. `SOFTSTART -> RUN`
   条件：相移缓升完成且无故障。

4. `RUN -> DERATING`
   条件：电流接近限值、母线压差过小、可达功率不足。

5. `DERATING -> RUN`
   条件：告警解除。

6. `RUN/DERATING -> FAULT`
   条件：过压、过流、参数非法、驱动异常。

7. `FAULT -> STOP`
   条件：故障清除且收到复位。

## 10. 推荐保护策略

## 10.1 最低限度必须有的保护

1. `Vdc1` 过压
2. `Vdc1` 欠压
3. `Vdc2` 过压
4. `Vdc2` 欠压
5. DAB 电流过流
6. 相移命令超限
7. 参数非法
8. 软启动超时

## 10.2 没有电流采样时的现实处理

当前仓库里没有明确的 DAB 电流反馈命名证据。

所以需要诚实地说：

**如果硬件上暂时没有 DAB 电流反馈，第一版只能做到“电压闭环 + 模型限幅 + 硬件级过流保护配合”，不能宣称自己有完整的内环电流控制。**

这时建议：

1. 软件侧保守限相移
2. 软件侧保守限相移斜率
3. 依赖硬件比较器/驱动故障做最后保护

## 11. 参数设计原则

## 11.1 必须显式注入的参数

这些参数不能偷用工程宏，必须作为配置显式给入：

1. `dab_fs_hz`
2. `dab_lk_h`
3. `dab_turn_ratio`
4. `dab_phi_max_rad`
5. `dab_phi_slew_rate`
6. `vdc2_ref`
7. `vdc2_kp`
8. `vdc2_ki`
9. `p_ref_max`
10. `p_ref_min`
11. `vdc1_uv_limit`
12. `vdc1_ov_limit`
13. `vdc2_uv_limit`
14. `vdc2_ov_limit`

原因很简单：

1. DAB 是新功率级，不应把参数偷偷绑死在现有 SVG 宏里。
2. 同样的控制算法将来可能移到别的额定值平台。

## 11.2 参数整定顺序

推荐顺序：

1. 先确认模型参数方向和量纲没错
2. 再确认 `phi_max`、`p_ref_max` 不会过冲
3. 再调软启动斜率
4. 最后再调 `Vdc2` 外环 PI

不要上来就调 PI。

因为很多 DAB “看起来像 PI 不稳”的问题，实际是：

1. 功率方向符号反了
2. 变比折算反了
3. 相移上限过大
4. 没有斜率限制

## 12. 推荐的系统协同关系

## 12.1 和并网侧的协同

加了 DAB 后，系统逻辑应该变成：

1. 并网侧负责把交流能量搬到 `Vdc1`
2. DAB 再把 `Vdc1` 的能量搬到 `Vdc2`
3. `Vdc2` 的扰动会传导到 `Vdc1`
4. 并网侧 `Vdc1` 环再把这个扰动从电网侧补回来

所以系统是两级串联调能：

```text
电网 <-> SVG 三相桥 <-> Vdc1 <-> DAB <-> Vdc2
```

### 关键原则

1. `Vdc1` 和 `Vdc2` 各有清晰“谁负责稳住它”的归属。
2. DAB 不能直接改三相补偿电流指令。
3. 并网侧也不应该直接产出 DAB 相移。

## 12.2 和通信/上位机的协同

上位机只应该看到“DAB 业务量”，而不是一堆桥臂寄存器细节。

推荐给上位机的量：

1. `dab_enable`
2. `dab_state`
3. `dab_fault`
4. `vdc1`
5. `vdc2`
6. `vdc2_ref`
7. `p_ref`
8. `phase_shift_cmd`
9. `phase_shift_actual`

## 13. 场景化输入 -> 输出

这里不只验证“能不能跑”，而是逐场景列逻辑。

## 13.1 场景 A：DAB 未使能

输入：

1. `enable = 0`
2. `Vdc1 = 正常`
3. `Vdc2 = 任意`

输出：

1. 状态 = `STOP`
2. `phase_shift_cmd = 0`
3. `p_ref = 0`
4. 故障字不一定置位，但运行位必须为 0

## 13.2 场景 B：`Vdc2` 低于参考，允许从 `Vdc1` 向 `Vdc2` 送能

输入：

1. `enable = 1`
2. `Vdc1` 正常
3. `Vdc2 < Vdc2_ref`
4. 无故障

输出：

1. 状态从 `READY -> SOFTSTART -> RUN`
2. `dab_vdc_ctrl` 输出正向 `P_ref`
3. `dab_phase_shift` 输出正向相移
4. `phase_shift_cmd` 逐步增大，不允许一步跳变
5. `Vdc2` 随时间上升并逼近参考

## 13.3 场景 C：`Vdc2` 高于参考，需要反向回能

输入：

1. `enable = 1`
2. `Vdc2 > Vdc2_ref`
3. 允许双向运行

输出：

1. `P_ref < 0`
2. `phase_shift_cmd < 0`
3. DAB 功率方向反向
4. `Vdc2` 被拉回参考值附近

## 13.4 场景 D：`Vdc1` 太低，不能继续向 `Vdc2` 送能

输入：

1. `Vdc1 < vdc1_uv_limit`
2. `Vdc2` 仍然低于参考

输出：

1. 不能因为 `Vdc2` 低就盲目加相移
2. 保护层应限制 `P_ref` 或直接闭锁
3. 状态进入 `DERATING` 或 `FAULT`
4. `phase_shift_cmd` 被压低或归零

## 13.5 场景 E：相移已经打满，但 `Vdc2` 仍上不去

输入：

1. `Vdc2` 长时间低
2. `phase_shift_cmd = phi_max`

输出：

1. 不允许积分继续无限累积
2. 电压环必须反积分饱和
3. 应上报“可达功率不足”或进入降额状态

## 13.6 场景 F：运行中故障触发

输入：

1. 运行中某一时刻 `Vdc2 > ov_limit` 或 DAB 电流过流

输出：

1. 状态跳到 `FAULT`
2. `phase_shift_cmd` 立即归零
3. 故障字锁存
4. 积分器冻结或复位

## 14. 选一个具体操作，从头到尾追踪

这里按“用户操作 -> 内部状态 -> 输出结果 -> 上位机看到什么”完整走一遍。

### 操作

用户在上位机上执行：

1. 使能 DAB
2. 设定 `Vdc2_ref = 760V`
3. 当前实测 `Vdc1 = 780V`
4. 当前实测 `Vdc2 = 710V`
5. 无故障

### 第一步：输入层拿到什么

算法库输入侧拿到：

1. `enable = 1`
2. `vdc_primary = 780V` 对应归一化值
3. `vdc_secondary = 710V` 对应归一化值
4. `vdc2_ref = 760V` 对应归一化值
5. `direction_mode = AUTO_BIDIRECTIONAL`

### 第二步：状态怎么变

1. `dab_mode` 看到使能有效、母线有效、参数合法。
2. 状态 `STOP -> READY`
3. 准备好后进入 `SOFTSTART`
4. 软启动期间允许输出一个受限的非零相移
5. 当软启动完成且无故障，进入 `RUN`

### 第三步：每个控制模块拿到什么数据

`dab_vdc_ctrl` 拿到：

1. `ref = 760V`
2. `fdb = 710V`
3. 误差为正

所以它输出：

1. 正的 `P_ref_raw`

`dab_power_feedforward` 拿到：

1. `P_ref_raw`
2. `Vdc1 = 780V`
3. `Vdc2 = 710V`
4. `Lk/n/fs`

所以它输出：

1. 经当前工况修正后的 `P_ref_limited`

`dab_phase_shift` 拿到：

1. `P_ref_limited > 0`
2. 当前功率增益
3. `phi_max`
4. `phi_slew_rate`

所以它输出：

1. 正向 `phi_ref`
2. 且这个 `phi_ref` 会被斜率限制成平滑增长

`dab_softstart` 拿到：

1. `phi_ref`
2. 当前还在 `SOFTSTART`

所以它输出：

1. 一个比 `phi_ref` 更保守的 `phi_cmd`

`dab_protection` 拿到：

1. `Vdc1` 正常
2. `Vdc2` 正常但偏低
3. 无过流

所以它输出：

1. 不触发闭锁

### 第四步：最终控制输出是什么

算法库最终给应用层的不是三相 `abc`，而是：

1. `phase_shift_cmd > 0`
2. `state = RUN` 或 `SOFTSTART`
3. `p_ref > 0`
4. `fault = 0`

### 第五步：应用层实际会做什么

应用层 `dab_pwm_adapter` 根据 `phase_shift_cmd`：

1. 计算主副桥 PWM 相对延时
2. 更新桥臂时序
3. 保证死区和互锁

### 第六步：上位机最终看到什么

上位机应该看到：

1. DAB 状态从“停止/就绪”切到“软启动/运行”
2. `Vdc2` 开始上升
3. `phase_shift_cmd` 从 0 平滑增大
4. 若系统正常，`Vdc2` 最终稳定在参考附近

## 15. 这份原理文档最终给 spec 什么结论

核心结论只有四条：

1. DAB 必须是**独立功率级控制模块**，不能并进现有三相 `current_loop`。
2. DAB 第一版建议采用 **SPS 单移相 + `Vdc2` 外环 + 功率前馈 + 软启动 + 保护层**。
3. 纯算法与硬件适配必须分层，PWM/ADC/共享内存/Modbus 继续留在应用层。
4. 新功能必须拆成小模块，而不是往一个 `dab.c` 里堆所有逻辑。
